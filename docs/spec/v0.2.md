# CxRP v0.2 — Contract × Request Protocol

CxRP v0.2 defines the message envelopes exchanged across OperatorConsole, SwitchBoard, and OperationsCenter. v0.1 is frozen; v0.2 is a breaking revision.

## Scope

CxRP standardizes payload structure, shared vocabulary, and validation format at repo boundaries. CxRP does **not** define runtime behavior, execution engines, scheduling, lane selection algorithms, adapters, queues, routing systems, or model invocation.

## Canonical Contracts

- `TaskProposal` — requested work intent.
- `LaneDecision` — chosen lane/executor/backend, produced by SwitchBoard.
- `ExecutionRequest` — bounded executable unit, produced by OperationsCenter.
- `ExecutionResult` — normalized completion payload.

## Layered Vocabulary

CxRP separates *category* from *specific implementation*:

- `lane: LaneType` — abstract category (`coding_agent`, `review_agent`, `local_model`, `hosted_model`).
- `executor: str | None` — specific tool within the category (e.g. `claude_cli`, `codex_cli`).
- `backend: str | None` — dispatch strategy (e.g. `direct_local`, `kodo`).

`executor` and `backend` are open strings at the CxRP envelope level. Consuming systems (e.g. OperationsCenter) layer their own `Literal` constraints internally without changing the wire contract.

## Per-Lane Payload Schemas

`ExecutionRequest.input_payload` is a free-form object at the envelope level; its inner shape is governed by a per-lane schema named in `ExecutionRequest.input_payload_schema`. Identifier format: `<name>/v<X.Y>` resolving to `cxrp/schemas/v<X.Y>/payloads/<name>.schema.json`.

Validation is two-stage: envelope validates against the contract schema, payload validates against the named payload schema.

Well-known payloads in v0.2:

- `coding_agent_input/v0.2` — repo, branches, workspace, validation commands.
- `coding_agent_target/v0.2` — repo coordinates for `TaskProposal.target`.

## Envelope Changes vs v0.1

- `LaneDecision` and `ExecutionRequest` add `executor`, `backend` (open strings).
- `LaneDecision.alternatives` is now a list of `LaneAlternative` objects (with their own `lane`, `executor`, `backend`, `confidence`, `reason`), not bare lane types.
- `ExecutionRequest` adds `input_payload_schema` and a typed `limits` object (`max_changed_files`, `timeout_seconds`, `require_clean_validation`). `constraints: list[str]` remains for natural-language guidance.
- `Artifact.artifact_type` (closed enum) is replaced by `Artifact.kind: str` (open). `ArtifactKind` enum (`input`, `output`, `log`, `diagnostic`) lists the canonical kinds; consumers may register additional kinds.
- `TaskProposal` adds optional `task_type`, `execution_mode`, `priority`, `risk_level`, `target` — all open at the envelope level.

## Vocabulary

- `LaneType` — `coding_agent`, `review_agent`, `local_model`, `hosted_model`.
- `ExecutionStatus` — `pending`, `accepted`, `running`, `succeeded`, `failed`, `cancelled`, `rejected`, `timed_out`.
- `ArtifactKind` — `input`, `output`, `log`, `diagnostic` (canonical only; field is open).

## Versioning

All envelopes carry `schema_version = "0.2"` and a `contract_kind` discriminator. Breaking envelope changes require a new `cxrp/schemas/vX.Y/`. v0.1 is frozen on disk for historical interop and remains under `cxrp/schemas/v0.1/`.
